# Commands

Reference for running genesis-env from your **project root** (current working directory).

## Default: interactive generate

Running genesis-env with **no subcommand** is the same as `generate`: it finds a template, prompts once per variable, then writes `.env`.

| Goal | Command |
|------|---------|
| No local install (recommended try) | `npx genesis-env` |
| After `npm i -g genesis-env` | `genesis-env` |
| After local `npm i genesis-env` | `npx genesis-env` or `node_modules/.bin/genesis-env` |

Template discovery, prompts, defaults, and overwrite rules are covered in [Templates & `.env`](/docs/validation).

## generate (explicit)

Same behavior as the default command:

```bash
npx genesis-env generate
```

Overwrite an existing `.env` instead of exiting with an error:

```bash
npx genesis-env generate --force
# or
npx genesis-env generate -f
```

You can also use the default form:

```bash
npx genesis-env --force
```

## init

Scaffold missing files and keep `.gitignore` safe:

```bash
npx genesis-env init
```

Creates missing `.env` and `.env.template` when needed, and ensures `.env` is listed in `.gitignore` (no duplicate lines).

## Options / flags

| Flag | Effect |
|------|--------|
| `--force` / `-f` | Overwrite an existing `.env` instead of exiting with an error (with `generate` or the default command). |

## Help

```bash
npx genesis-env --help
```

## Sample workflow

1. `npx genesis-env init` (optional) — scaffold `.env`, `.env.template`, update `.gitignore` if needed.  
2. Edit `.env.template` or `.env.example` with your variable names and optional defaults.  
3. `npx genesis-env` — answer prompts to generate `.env`.  
4. Run your app with the new `.env`.

For template file rules and examples, see [Templates & `.env`](/docs/validation).
